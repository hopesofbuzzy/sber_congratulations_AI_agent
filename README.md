# Sber Congratulations AI Agent (MVP)

MVP-проект: **автоматизированный конвейер подготовки и доставки персонализированных поздравлений** клиентам/организациям Сбера на основе событий (ДР, праздники, памятные даты, ручные триггеры), с веб-интерфейсом для контроля статусов, ручных действий и последующего развития.

## Репозиторий

- **CI**: GitHub Actions workflow в `.github/workflows/ci.yml` (ruff/black/pytest)
- **Ключевые проектные документы**: начните с `docs/PROJECT_OVERVIEW.md` и `docs/DECISIONS.md`

## Что входит в MVP

- **Регулярный/событийный триггер**: поиск ближайших событий (на N дней вперёд) + ручной запуск сотрудником.
- **Извлечение данных**: базовая CRM-модель (клиенты/контакты/сегменты/история) + праздники.
- **Обогащение организаций (MVP+)**:
  - хранение `ИНН`, официального названия компании, `ОКВЭД`, руководителя, сайта и источника,
  - импорт условной базы компаний из CSV в выделенной директории ресурсов,
  - bulk/manual enrichment через `demo`, `dadata` или `hybrid` провайдер,
  - статусы `not_requested/pending/enriched/error` для контроля качества данных.
- **Генерация контента**:
  - выбор тона/шаблона по типу события и сегменту,
  - персонализация (имя/отчество/компания/должность/отрасль; без выдумывания фактов и без утечек тем последних контактов),
  - генерация «открытки» (картинка) в виде файла,
  - event-specific image presets: день рождения, Новый год, 8 Марта и деловые поводы используют разные визуальные мотивы при сохранении общих запретов на людей и текст,
  - более строгий JSON-контракт для GigaChat-текста, чтобы уменьшить число fallback-сценариев из-за “почти JSON”.
- **VIP approval**: для клиентов сегмента `vip` поздравление создаётся со статусом `needs_approval` и **отправляется только после подтверждения** в UI.
- **Доставка**: MVP-канал отправки в **outbox (файлы)** + лог доставки (идемпотентность).
- **SMTP email rendering**: при реальной email-отправке используется branded HTML-письмо в зелёной теме Сбера с акцентной шапкой, inline-иллюстрацией и plain-text fallback; в самом письме оставлен только клиентский контент без технических пояснений про внутренний пайплайн.
- **Demo-friendly delivery fallback**: если включён `smtp`, но у demo/импортированного клиента нет пригодного email, система не валит прогон в ошибку, а безопасно пишет доставку в file-outbox.
- **Аудит запусков агента**: каждый запуск фиксируется как `AgentRun`, отображается в UI и теперь имеет детальную страницу, где видно, какие именно поздравления, доставки и feedback связаны с конкретным прогоном.
- **Постоянное улучшение**: менеджерский feedback по поздравлениям (оценка/исход/комментарий) сохраняется в БД и может использоваться для дальнейшего анализа качества.
- **Веб-интерфейс**: просмотр клиентов/событий/поздравлений/доставок, создание клиента, ручной триггер, запуск агента.
- **Ручные кампании для реальной базы**: можно создать единичное ручное событие или быструю demo-кампанию для импортированных клиентов, чтобы агент сгенерировал поздравления не только по ДР/праздникам.
- **Тесты**: базовая проверка детектора событий и идемпотентной отправки.

## Архитектура (упрощённо)

1) **Detector** находит события на горизонте \(N\) дней: ДР клиентов + праздники из справочника.  
2) **Agent Orchestrator** собирает контекст, выбирает тон/шаблон, генерирует текст/картинку.  
3) **Sender** доставляет через канал (в MVP: файловый outbox), фиксирует Delivery с **idempotency_key**.  
4) **Web/UI + API** даёт контроль и ручные операции.

Ключевой принцип доставки настраивается через `.env`:  
- `DELIVERY_SCHEDULE_MODE=event_date` — поздравления могут генерироваться заранее, но отправка происходит только в день события;  
- `DELIVERY_SCHEDULE_MODE=immediate` — поздравление отправляется сразу после генерации/approve.  
Для VIP в обоих режимах требуется ручной approve.

## Скрипты (Windows)

- **`scripts/setup_backend.cmd`**: создаёт `backend/.venv`, ставит зависимости, создаёт `backend/.env` из `backend/env.example`.
- **`scripts/run_backend.cmd`**: запускает web UI + API (по умолчанию порт **8001**; можно переопределить через `PORT`).
- **`scripts/run_scheduler.cmd`**: запускает планировщик (демо «регулярного» режима).
- **`scripts/kill_port.cmd`**: освобождает порт (если после прошлых запусков остались «залипшие» процессы).

## Быстрый старт (локально, Windows)

### 1) Требования

- **Python 3.10+** (MVP не требует 3.11; важно для Windows-демо)

### 2) Установка

Самый простой вариант:

```bat
scripts\setup_backend.cmd
```

Либо вручную:

```bat
cd backend
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt -r requirements-dev.txt
copy env.example .env
```

### 3) Запуск веб-интерфейса + API

```bat
cd ..
scripts\run_backend.cmd
```

По умолчанию используется порт **8001**, но на некоторых Windows-окружениях отдельные порты могут быть запрещены (WinError 10013).
Скрипт запуска в этом случае **автоматически подберёт ближайший доступный порт** и выведет его в консоль.

Если нужен конкретный порт (предпочтительный):

```bat
set PORT=8000
scripts\run_backend.cmd
```

Если порт “залип” после прошлых запусков:

```bat
scripts\kill_port.cmd 8000
```

### 4) Демо-сценарий (рекомендуемый)

1. В UI нажмите **Seed demo data** (слева) — он **пересоздаст** демо-набор из **5 случайных клиентов**.
   Дни рождения выбираются так, чтобы ближайшие события были **сегодня или в ближайшие дни** (без “прошлого”).
2. При необходимости нажмите **Импортировать базу компаний**, затем **Enrich company profiles**: система заполнит организационные профили через выбранный провайдер (`demo`/`dadata`/`hybrid`).
3. Для реальной базы перейдите в **Events** и создайте ручное событие или нажмите **Подготовить события для реальной базы**.
4. Нажмите **Run agent now** (на дашборде): агент создаст поздравления на ближайшие события.
5. Перейдите в **Greetings / Deliveries** и проверьте статусы.
6. На странице **Greetings** сохраните feedback менеджера и покажите, что он сохраняется в системе.
7. Посмотрите «отправленные» материалы:
   - `backend\data\outbox\` (файлы сообщений),
   - `backend\data\cards\` (сгенерированные открытки).

## Переменные окружения

См. `backend/env.example`. Важное:
- **LOOKAHEAD_DAYS**: горизонт поиска событий.
- **MAX_HOLIDAY_RECIPIENTS**: лимит получателей на один праздник (чтобы не сжигать токены на демо).
- **MAX_GIGACHAT_IMAGES_PER_RUN**: лимит генераций изображений через GigaChat за один прогон (скорость/токены). Остальные картинки — Pillow fallback.
- **GIGACHAT_IMAGE_GENERATION_TIMEOUT_SEC**: отдельный таймаут именно на генерацию изображения через GigaChat; полезно повышать для демо, если часть картинок не успевает вернуться.
- **SEND_MODE**: `file` (MVP) — пишет письма в outbox.
- **DELIVERY_SCHEDULE_MODE**: `event_date` или `immediate`.
- **OUTBOX_DIR**: куда «отправлять».
- **COMPANY_ENRICHMENT_PROVIDER**: `demo`, `dadata`, `hybrid`.
- **COMPANY_IMPORT_CSV_PATH**: путь к CSV-справочнику компаний.
- **DADATA_API_KEY**: ключ для `find-party` в DaData.
- **LLM_MODE**: `template` (по умолчанию, офлайн) или `openai` (OpenAI-compatible HTTP API).
- **IMAGE_MODE**: `pillow` (по умолчанию) или `gigachat` (генерация открыток через GigaChat).

## Что добавлено сверх исходного MVP

- **Company enrichment layer**: профиль клиента теперь может хранить `ИНН`, официальное название компании, `ОКВЭД`, руководителя, сайт и `source_url`.
- **Provider-based enrichment**: для презентации работает локальный demo-registry, а для реального обогащения можно включить `DaData` через `.env`.
- **Company import flow**: условная база компаний лежит в `backend/app/resources/company_data/` и может быть импортирована в UI/API без ручного копирования в корень.
- **Manual event flow for real clients**: страница `Events` позволяет создать управляемые поводы для импортированной базы, чтобы генерация не зависела только от дней рождения и календарных праздников.
- **Holiday knowledge base**: встроенный каталог праздников и проф-поводов теперь хранит не только названия, но и смысловые теги (`category`, `focus_hint`, `prompt_hint`), чтобы генерация лучше различала типы поздравительных сценариев.
- **Semantic prompt-building**: текстовые и image-промпты начали собираться через общий semantic-layer события, а не только через набор точечных условий по названию праздника.
- **Improved personalization**: fallback и LLM-промпты используют отраслевой контекст (`okved_name`) и официальное название компании.
- **Feedback loop**: менеджер может на странице `Greetings` сохранить `score/outcome/notes` для каждого поздравления.
- **Dashboard metrics**: на главной странице видны счётчики enriched-клиентов и feedback-записей.
- **Pipeline funnel on dashboard**: на главной странице видно post-generation воронку `generated -> needs approval -> delivered -> feedback` и быстрые индикаторы проблемных запусков/ошибок доставки.
- **Run-level audit**: страница `Runs` теперь ведёт в детализацию конкретного запуска и показывает не только агрегаты, но и созданные им поздравления.
- **Presentation-ready UI refresh**: обновлены общая оболочка интерфейса, KPI-карточки, навигация, таблицы и сценарная подача страниц `Dashboard/Clients/Events/Greetings/Deliveries/Runs`.

### GigaChat (опционально)

Для интеграции с GigaChat укажите:
- `LLM_MODE=gigachat` и/или `IMAGE_MODE=gigachat`
- `GIGACHAT_CREDENTIALS` (Authorization Key)

Важно: для TLS может понадобиться корневой сертификат Минцифры (см. `instruction_gigachat.md`). В крайнем случае для демо можно временно отключить проверку сертификата: `GIGACHAT_VERIFY_SSL_CERTS=false`.


## (Опционально) Планировщик

Если хотите показать «регулярный» режим, можно запустить планировщик (cron daily 09:00):

```bat
scripts\run_scheduler.cmd
```

Регулярный режим делает одно и то же каждый день:
- создаёт/обновляет события на горизонт `LOOKAHEAD_DAYS`
- генерирует поздравления заранее
- **отправляет только те поздравления, у которых дата события = сегодня** (VIP — только после approve)

Для удобства демо планировщик также делает **один запуск сразу при старте**, чтобы не ждать 09:00.

# Team Workflow

Краткая и практичная инструкция, как организовать совместную разработку по проекту без хаоса.

## 1. Главные правила

1. Не работать напрямую в `main`.
2. Одна задача = одна ветка.
3. Один законченный функциональный блок = один `PR`.
4. Перед merge обязательно: `pytest`, `ruff`, `black --check`.
5. Если изменилось поведение системы, обновляем документацию в этом же блоке.
6. Если задача задевает несколько подсистем, у неё должен быть один ответственный интегратор.
7. Нельзя “заодно” переписывать чужую часть проекта без отдельного согласования.
8. Любое спорное архитектурное решение сначала фиксируем в обсуждении, потом кодим.

## 2. Как делить проект

Проект делится не “по людям вообще”, а по подсистемам.

### Backend / Delivery / DB
- Зона ответственности: модели, статусы, доставка, маршруты, миграции.
- Основные файлы: `backend/app/db/*`, `backend/app/api/*`, `backend/app/services/sender.py`, `backend/app/services/due_sender.py`, `backend/app/services/approval.py`.

### AI / Agent / Prompt-building
- Зона ответственности: оркестратор, генерация текста, prompt-building, image prompts, GigaChat.
- Основные файлы: `backend/app/agent/*`.

### Web / UX / Demo-flow
- Зона ответственности: страницы, формы, dashboard, сценарий демонстрации.
- Основные файлы: `backend/app/web/*`.

### Data / Enrichment / Events
- Зона ответственности: импорт, enrichment, DaData, knowledge layer праздников и manual events.
- Основные файлы: `backend/app/services/company_*`, `backend/app/services/dadata_client.py`, `backend/app/services/holiday_catalog.py`, `backend/app/services/event_detector.py`, `backend/app/services/manual_events.py`, `backend/app/resources/*`.

### QA / Docs / Release hygiene
- Зона ответственности: тесты, CI, документация, проверка demo-ready состояния.
- Основные файлы: `backend/tests/*`, `.github/workflows/*`, `README.md`, `docs/*`, `backend/env.example`.

## 3. Как работать через Git

### Базовая схема
1. `main` — только стабильная ветка.
2. Под каждую задачу создаётся отдельная ветка.
3. Работа идёт в своей ветке.
4. Потом создаётся `PR`.
5. После проверки и тестов изменения вливаются в `main`.

### Нормальные имена веток
- `feat/manual-events-ui`
- `feat/dadata-batch-update`
- `fix/gigachat-json-output`
- `fix/smtp-fallback`
- `docs/team-workflow`

### Нормальные коммиты
- `feat(agent): add event-specific image presets`
- `fix(sender): fallback to file when email is missing`
- `docs(workflow): add team collaboration rules`

## 4. Как работать через GitHub

### Минимально обязательные сущности
- `Issues` — список задач.
- `Pull Requests` — проверка изменений.
- `Projects` board — статусы задач.

### Минимальные статусы задач
- `Todo`
- `In Progress`
- `Review`
- `Done`

### Что должно быть у каждой задачи
- что делаем;
- зачем;
- какие файлы/подсистемы затрагиваем;
- как проверяем результат.

## 5. Как выглядит нормальный рабочий цикл

1. Создали `Issue`.
2. Назначили ответственного.
3. Создали ветку от `main`.
4. Сделали задачу маленьким законченным блоком.
5. Прогнали локально:
   - `pytest`
   - `ruff`
   - `black --check`
6. Обновили документацию, если поведение изменилось.
7. Создали `PR`.
8. Другой участник посмотрел изменения.
9. После проверки влили в `main`.

## 6. Кто за что отвечает в команде

У каждой подсистемы есть основной ответственный. Это не значит, что только он может менять код. Это значит, что:
- он главный по смыслу;
- он проверяет спорные изменения;
- он помогает не ломать архитектуру своей зоны.

## 7. Что нельзя делать

- коммитить прямо в `main`;
- вести несколько несвязанных задач в одной ветке;
- смешивать UI, AI, docs и случайные bugfix в один PR без причины;
- изменять чужую подсистему “по пути” без согласования;
- считать задачу завершённой без проверки и без обновления docs, если поведение изменилось.

## 8. Практичный режим именно для нашей команды

Если команда только начинает работать нормально, то достаточно такой схемы:
- один `Issue` = одна задача;
- одна ветка = одна задача;
- один `PR` = один логический блок;
- минимум один просмотр со стороны другого участника;
- merge только после локальных проверок.

Это уже выглядит профессионально и резко снижает хаос.

## 9. С чего начать прямо сейчас

1. Зафиксировать `main` как стабильную ветку.
2. Разбить текущий backlog по 5 зонам ответственности.
3. Завести `Issues` на ближайшие реальные задачи.
4. Назначить owner на каждую задачу.
5. Начать работать только через feature-ветки и `PR`.

Готовый стартовый набор задач уже подготовлен в `docs/ISSUES_BOOTSTRAP.md`.

## 10. Главное

Нормальная совместная разработка — это не сложный enterprise-процесс.  
Это простая дисциплина:

- понятные роли,
- понятные границы файлов,
- понятные ветки,
- понятные проверки,
- и единый порядок работы для всех.

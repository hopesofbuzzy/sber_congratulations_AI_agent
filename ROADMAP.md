# ROADMAP

Документ фиксирует текущее состояние проекта, распределение ответственности между
подсистемами и приоритеты дальнейшего развития.

## Цель проекта

Развивать систему автоматизированной подготовки и доставки персонализированных
поздравлений для клиентских сценариев:

- получение и нормализация данных о клиенте и компании;
- формирование повода для коммуникации;
- генерация текста и иллюстрации;
- доставка, контроль статусов и аудит результата;
- получение операторской обратной связи.

## Текущее состояние

На текущем этапе в проекте реализованы следующие базовые контуры:

1. **Данные и профили клиентов**
   - хранение клиентских данных, событий, поздравлений, доставок и feedback;
   - импорт клиентской базы из CSV;
   - enrichment профиля компании через локальный источник и `DaData`;
   - хранение организационных атрибутов: `ИНН`, `ОКВЭД`, руководитель, сайт, источник данных.

2. **Генерация контента**
   - template fallback для офлайн-режима;
   - интеграция с `GigaChat` для генерации текста и изображений;
   - семантический слой для событий и поводов;
   - управляемые ограничения на формат ответа и сценарии fallback.

3. **Операторский контур**
   - веб-интерфейс для работы с клиентами, событиями, поздравлениями и доставками;
   - ручные события для импортированной базы;
   - approve-flow для клиентов категории `vip`;
   - аудит запусков агента через `AgentRun`.

4. **Доставка и контроль качества**
   - file outbox как базовый безопасный канал;
   - SMTP-отправка с HTML-письмом и plain-text fallback;
   - защита от ошибочной доставки при неполных контактах;
   - тесты, CI и конфигурация окружения.

## Распределение ответственности

Проект следует делить не по абстрактным “задачам вообще”, а по подсистемам и
файлам.

### 1. Backend / Data Model

- Зона ответственности: модели данных, API, статусы, жизненный цикл сущностей, доставка, approve-flow.
- Основные файлы:
  - `backend/app/db/*`
  - `backend/app/api/*`
  - `backend/app/services/sender.py`
  - `backend/app/services/due_sender.py`
  - `backend/app/services/approval.py`

### 2. Генерация контента и интеграции LLM

- Зона ответственности: orchestration, prompt-building, fallback-логика, текстовая и визуальная генерация, правила безопасности генерации.
- Основные файлы:
  - `backend/app/agent/*`

### 3. Web / UI

- Зона ответственности: интерфейс оператора, маршруты web-слоя, формы, таблицы, сценарии ручного управления.
- Основные файлы:
  - `backend/app/web/*`

### 4. Data / Enrichment

- Зона ответственности: импорт данных, enrichment, календарь и классификация поводов, ручные события, нормализация профилей.
- Основные файлы:
  - `backend/app/services/company_*`
  - `backend/app/services/dadata_client.py`
  - `backend/app/services/holiday_catalog.py`
  - `backend/app/services/event_detector.py`
  - `backend/app/services/manual_events.py`
  - `backend/app/resources/*`

### 5. QA / Infrastructure

- Зона ответственности: тесты, CI, конфигурация окружения, воспроизводимый запуск, эксплуатационная документация.
- Основные файлы:
  - `backend/tests/*`
  - `.github/workflows/*`
  - `README.md`
  - `docs/*`
  - `backend/env.example`

## Принципы развития

- Изменения должны вноситься по подсистемам, а не случайными смешанными пакетами.
- Новая функциональность должна сопровождаться проверяемым сценарием использования.
- Конфигурация должна выноситься в `.env`, а не зашиваться в код.
- Для внешних интеграций обязателен безопасный fallback.
- Любые изменения пользовательского поведения должны сопровождаться обновлением документации.
- Для критичных сценариев должны поддерживаться автоматические проверки.

## Ближайшие приоритеты

### 1. Стабилизация операционного контура

- выровнять жизненный цикл `Greeting`, `Delivery`, `AgentRun` и feedback;
- сократить число неочевидных fallback-сценариев;
- обеспечить предсказуемое поведение в `file` и `smtp` режимах.

### 2. Качество генерации

- повысить стабильность текстовой генерации;
- расширять правила персонализации на основе реальных данных профиля;
- улучшать различимость сценариев генерации по типам событий;
- ограничивать ошибки формата и неуправляемую вариативность внешней модели.

### 3. Качество входных данных

- развивать импорт клиентской базы;
- улучшать enrichment и обработку ошибок внешних источников;
- расширять классификацию календарных и профессиональных поводов;
- улучшать нормализацию сегмента, роли клиента и организационного контекста.

### 4. Интерфейс и операторские сценарии

- упрощать путь оператора от загрузки данных до результата;
- поддерживать консистентность интерфейса и читаемость статусов;
- улучшать связанные сценарии `Clients -> Events -> Greetings -> Deliveries -> Runs`.

### 5. Контроль качества и сопровождение

- поддерживать `pytest`, `ruff`, `black --check` как обязательный минимум;
- расширять покрытие на критичных сценариях доставки, enrichment и генерации;
- удерживать документацию в соответствии с актуальным состоянием проекта.

## Среднесрочное развитие

### Интеграции

- подключение дополнительных источников клиентских данных;
- расширение каналов доставки;
- развитие работы с реальными корпоративными системами и ограничениями доступа.

### Аналитика и feedback

- накопление и анализ обратной связи;
- улучшение критериев качества поздравлений;
- развитие метрик по доставке, согласованию и отклику.

### Надёжность и эксплуатация

- развитие наблюдаемости и диагностических отчётов;
- усиление проверки конфигурации и безопасных режимов запуска;
- упрощение локального и командного воспроизведения среды.

## Дальнейшее расширение

В перспективе проект может развиваться в направлениях:

- более глубокая интеграция с CRM и корпоративными справочниками;
- новые каналы доставки;
- расширение правил генерации на более широкий набор клиентских сценариев;
- более строгие требования по безопасности, доступам и хранению данных.
